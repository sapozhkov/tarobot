# Iteration Plan

План ближайших итераций без Telegram.

Предпосылка:

- Telegram считаем последним этапом доставки;
- сейчас он не влияет на проверку основной гипотезы;
- главная цель ближайших итераций: превратить локальный `na-kolenke MVP` в устойчивый технический контур, который можно потом подключить к Raspberry Pi и механике.

## Где мы сейчас

Уже есть рабочий локальный baseline:

- вход: текстовый вопрос;
- simulated draw на 3 или 5 карт;
- локальная генерация текста через `MockLLMService`;
- локальный TTS через macOS `say`;
- сохранение артефактов на диск.

Это хорошо как proof-of-concept, но пока не проверяет самое важное:

- реальную работу с фотографией карт;
- ошибки физической раскладки;
- качество распознавания карты;
- качество голоса;
- качество реальной LLM-генерации;
- управляемость сценария гадания;
- границу между облачной частью и локальным агентом рядом с железом.

## POC gates

Для текущего POC первоочередными проверками считаем:

- `R1`: система находит все карты на фото;
- `R3`: система не врет про распознанную карту;
- `R9`: итоговый голосовой артефакт звучит достаточно живо.

Пока эти три проверки не пройдены, остальные итерации считаются вторыми по приоритету.

## Зафиксированное уточнение по vision

POC на обычных игральных картах подтвердил полезность текущего detection/count/error-handling контура, но не подтвердил corner-based распознавание как правильное финальное решение для Таро.

Дальше считаем целевым направлением:

- detection и rectification как общий reusable слой;
- распознавание Таро по всей карте;
- явное определение `upright / reversed`;
- OCR подписи как fallback или second signal;
- отдельный слой интерпретации layout для разных типов раскладов.

## Ближайшие итерации

### Итерация 1. Vision MVP: найти карты и понять, что расклад валиден

Цель:

Научиться по фотографии понимать, что карты действительно разложены, что их нужное количество, и что их вообще можно дальше распознавать.

Что делаем:

- фиксируем фото-стенд для первых экспериментов:
  - камера сверху;
  - контролируемый фон;
  - стабильный свет;
- вводим сущность `CaptureResult`;
- добавляем quality gate перед распознаванием:
  - кадр не пустой;
  - кадр не слишком смазан;
  - освещение в допуске;
- реализуем поиск card candidates на фото;
- реализуем проверку количества найденных карт против ожидаемого;
- сохраняем debug artifacts:
  - исходное фото;
  - фото с рамками;
  - координаты найденных карт;
  - reason code при ошибке.

Критерий успеха:

По фото система может уверенно ответить: "вижу нужное количество карт", "вижу не все карты" или "снимок невалидный".

### Итерация 2. Vision MVP: распознавание карты и orientation

Цель:

Научиться по найденным картам определять, какая именно карта выпала и в каком положении она лежит.

Что делаем:

- выравниваем найденную карту в нормализованный crop;
- определяем orientation;
- реализуем распознавание карты по всей поверхности, а не по углам;
- добавляем библиотеку эталонов конкретной колоды Таро;
- проверяем карту в normal и `180°`;
- используем OCR подписи как дополнительный сигнал;
- вводим confidence score;
- добавляем набор reason codes:
  - `card_not_found`;
  - `cards_count_mismatch`;
  - `card_low_confidence`;
  - `capture_invalid`;
  - `cards_overlap_suspected`;
- проверяем consistency rules:
  - нет ли невозможных дублей;
  - совпадает ли число распознанных карт с ожиданием;
  - не слишком ли низкая суммарная уверенность.

Критерий успеха:

На контролируемом фото-стенде система стабильно выдает список карт с confidence и умеет честно отказываться, если уверенности недостаточно.

### Итерация 3. TTS pass: проверить "живость" аудио

Цель:

Понять, дает ли система голосовой артефакт, который действительно поддерживает атмосферу сервиса.

Что делаем:

- вводим интерфейс `SpeechSynthesizer`;
- разделяем текст на сегменты:
  - вступление;
  - интерпретация карт;
  - финальный вывод;
- пробуем минимум два режима:
  - локальный `say`;
  - внешний TTS provider, если появится ключ;
- добавляем нормализацию текста под озвучку:
  - паузы;
  - короткие предложения;
  - более разговорный синтаксис;
- делаем ручную оценку 5-10 аудио-артефактов.

Критерий успеха:

Получается аудио, которое уже можно показывать как заготовку под "голос гадалки", а не только как технический артефакт.

### Итерация 4. Реальная LLM вместо mock

Цель:

Проверить, что сервис умеет давать уже не шаблонный, а реально интересный и управляемый текст гадания.

Что делаем:

- вводим интерфейс `ReadingGenerator`;
- оставляем `MockLLMService` как fallback;
- добавляем реальный LLM adapter;
- выносим prompt-шаблоны в отдельные файлы или модуль;
- делаем структурированный выход:
  - краткое summary;
  - блок по каждой карте;
  - финальный advice;
  - отдельный текст для озвучки;
- добавляем конфиг через env vars;
- фиксируем 10-20 тестовых вопросов для ручной оценки качества.

Критерий успеха:

По одному и тому же набору карт система выдает стабильно вменяемые, связные и достаточно "атмосферные" тексты.

### Итерация 5. Доменная модель ритуала

Цель:

Собрать минимальную orchestration-модель до железа, чтобы потом не переписывать все вокруг механики.

Что делаем:

- вводим сущность `Session`;
- описываем минимальные шаги пайплайна:
  - `accepted`;
  - `cards_drawn`;
  - `reading_generated`;
  - `speech_generated`;
  - `artifact_ready`;
- добавляем `session.json` и event log;
- делаем idempotent-поведение для повторного запуска;
- добавляем режим `replay` для существующей сессии.

Критерий успеха:

Один и тот же запуск можно повторять, доразбирать и перегенерировать без ручной возни с файлами.

### Итерация 6. Контракт с Raspberry Pi и hardware simulator

Цель:

Отделить "мозги" от "железа" и заранее определить, как облачная часть будет разговаривать с локальным агентом.

Что делаем:

- описываем протокол между orchestrator и `hardware-agent`;
- определяем команды:
  - `ping`;
  - `shuffle`;
  - `deal`;
  - `capture`;
  - `collect`;
  - `status`;
  - `stop`;
- определяем ответы:
  - `accepted`;
  - `in_progress`;
  - `completed`;
  - `failed`;
- делаем локальный simulator этого протокола;
- проверяем таймауты и повторные команды.
- фиксируем реакции на photo-related ошибки:
  - повторный снимок;
  - повторный расклад;
  - аварийное завершение с причиной.

Критерий успеха:

Сервис может работать не только с локальной функцией, но и с отдельным процессом, который выглядит как будущий Raspberry Pi agent.

### Итерация 7. Raspberry Pi agent stub

Цель:

Запустить простой технический агент, который уже можно поднять рядом с железом, даже если механика еще не готова.

Что делаем:

- делаем отдельный сервис `hardware-agent`;
- добавляем healthcheck;
- добавляем командный API;
- добавляем очередь локальных hardware jobs;
- добавляем логирование и сохранение результатов команд;
- подключаем simulator adapter.

Критерий успеха:

Можно разнести систему на две части:

- основной сервис;
- локальный агент на Raspberry Pi.

Даже без настоящей механики они уже могут взаимодействовать в реалистичном режиме.

## Что нам сейчас не хватает

### Критично для ближайшего движения

- photo capture contract;
- pipeline поиска карт на фото;
- confidence-модель распознавания;
- quality pass по TTS;
- policy для ошибок раскладки и распознавания;
- реального LLM adapter;
- конфигурации через env vars;
- нормального разделения интерфейсов `reading generation` / `tts` / `hardware`;
- набора тестовых сценариев для оценки качества гадания;
- session/event модели вместо просто папки с файлами;
- контракта общения с Raspberry Pi agent.

### Нужно скоро, но не прямо сейчас

- минимального safety layer;
- нормального логирования;
- базовой типизации и линтинга;
- фикстур для воспроизводимых раскладов;
- абстракции под внешний TTS provider;
- собственного набора фото с реальной раскладкой;
- формата артефактов, который потом не стыдно отдавать наружу.

### Пока можно не трогать

- Telegram;
- полноценную очередь задач;
- видео;
- продвинутую админку;
- платежи и прочую продуктовую обвязку.

## Рекомендуемый порядок прямо сейчас

Если идти прагматично, я бы делал так:

1. Зафиксировать photo capture условия и собрать первые реальные фото.
2. Сделать detection/count/quality gate.
3. Сделать card recognition + confidence + failure policy.
4. Сделать TTS pass и проверить "живость" аудио.
5. Подключить реальную LLM.
6. Ввести `Session` и event log.
7. Описать и собрать `hardware-agent` simulator с отдельным API.
8. Только потом идти в реальный Raspberry Pi.

## Что станет результатом этих итераций

После этих шагов у нас будет не просто демо-скрипт, а уже нормальный инженерный контур:

- валидация физической раскладки по фото;
- распознавание карт с контролем ошибок;
- голосовой артефакт, который поддерживает атмосферу;
- реалистичный текст;
- воспроизводимые сессии;
- понятная граница между сервисом и железом;
- возможность подключать настоящую механику без большого переписывания.
