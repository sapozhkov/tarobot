# Tarobot

Сервис автоматизированного гадания на картах Таро с выдачей результата через Telegram.

Проект состоит из двух частей:

- программная часть, которая живет в этом репозитории;
- механическая часть, которая собирается отдельно, но управляется из этого же контура через программный интерфейс.

## Зачем мы это делаем

Быстро проверить гипотезу, что можно собрать end-to-end сервис, который:

- принимает запрос пользователя в Telegram;
- безопасно валидирует запрос;
- формирует вступление и итоговое гадание с помощью ИИ;
- запускает физический ритуал: перемешивание, расклад, съемка карт, сбор карт;
- собирает артефакты ответа: видео, аудио, фото карт, текст;
- отправляет результат пользователю и закрывает задачу.

## Ключевая идея

Для первого этапа нужно проектировать систему так, чтобы она работала даже без готовой механики.

То есть программная часть должна уметь:

- работать через симулятор механики;
- работать с предзаписанным видео или через программную сборку ролика;
- подставлять карты вручную или из тестового набора;
- позднее без перелома архитектуры переключаться на реальную камеру и реальный контроллер механики.

Это главный принцип для быстрого proof of concept.

## Верхнеуровневый flow

1. Telegram-бот принимает заявку.
2. Заявка попадает в очередь.
3. Выполняется валидация:
   - запрещенные темы;
   - prompt injection;
   - техническая корректность запроса;
   - лимиты и антиспам.
4. Заявка переводится в `ready`.
5. Оркестратор берет заявку в работу.
6. Запускается запись видео или создается план сборки итогового видео.
7. LLM формирует:
   - понимание запроса;
   - вводную часть;
   - структуру будущего ответа.
8. Генерируется TTS для вступления.
9. Вступление проигрывается физически или накладывается на видео.
10. Во время вступления запускается механика перемешивания колоды.
11. Карты физически раскладываются.
12. Снимается фото или серия фото карт.
13. Выполняется распознавание карт.
14. LLM генерирует текст гадания на основе запроса и выпавших карт.
15. Генерируется TTS для итогового ответа.
16. Записывается или собирается финальный видеорезультат.
17. Карты физически собираются.
18. Формируются артефакты ответа:
   - видео;
   - аудио;
   - фото карт;
   - текст гадания.
19. Ответ отправляется пользователю.
20. Заявка переводится в `done` или `failed`.

## Реализуемо ли это

Да, реализуемо.

Основная сложность не в одной "магической" технологии, а в правильной декомпозиции на независимые контуры:

- прием и безопасная обработка заявок;
- оркестрация долгих задач;
- генерация текста и голоса;
- видео-конвейер;
- CV-распознавание карт;
- управление механикой;
- надежное восстановление после сбоев.

Самый важный вывод: не нужно ждать готовой механики, чтобы начать проверять гипотезу. Почти весь риск можно снять на программном и полу-ручном стенде.

## Рекомендуемая архитектура для POC

Для быстрого старта выглядит разумно начать с Python-монорепозитория.

Почему Python:

- удобно для Telegram-ботов;
- удобно для очередей и воркеров;
- удобно для computer vision;
- удобно для интеграции с железом;
- проще и быстрее для POC, чем раскладывать систему по разным языкам.

### Компоненты

1. `bot`
   Принимает запросы из Telegram, показывает статусы, отправляет итоговые артефакты.

2. `api`
   Сервис для внутренних команд, healthcheck, админки и вебхуков, если они понадобятся.

3. `orchestrator`
   Ведет состояние заявки и вызывает шаги пайплайна в нужном порядке.

4. `workers`
   Исполняют отдельные фоновые задачи:
   - moderation/validation;
   - LLM;
   - TTS;
   - media/render;
   - vision;
   - delivery.

5. `hardware-agent`
   Отдельный адаптер для общения с физическим устройством.
   Может жить на той же машине, что управляет камерами и контроллерами.

6. `storage`
   Хранение артефактов и временных файлов:
   - исходные снимки;
   - финальные видео;
   - аудио;
   - логи шагов;
   - отладочные данные.

7. `backoffice`
   Минимальный интерфейс оператора:
   - видеть очередь;
   - повторить шаг;
   - поставить заявку на ручную проверку;
   - отметить сбой механики;
   - перевести в retry или failed.

### Принцип интеграции с механикой

Механика должна быть спрятана за абстракцией.

Например, доменный интерфейс может выглядеть так:

- `shuffle_deck()`
- `deal_cards(spread_spec)`
- `capture_layout()`
- `collect_cards()`
- `get_status()`
- `emergency_stop()`

Тогда у нас будет минимум два адаптера:

- `simulator`, который нужен для разработки и тестов;
- `real_device`, который позже подключится к настоящей механике.

## Предлагаемая модель состояний заявки

Черновой state machine:

- `created`
- `queued`
- `validating`
- `rejected`
- `ready`
- `running`
- `intro_generating`
- `intro_playing`
- `shuffling`
- `dealing`
- `capturing`
- `recognizing`
- `reading_generating`
- `rendering`
- `delivering`
- `done`
- `failed`
- `manual_review`

Важно, чтобы каждый шаг был:

- идемпотентным;
- логируемым;
- возобновляемым после падения;
- понятным для оператора.

## Что лучше сделать в POC, а что отложить

### В POC обязательно

- Telegram intake;
- очередь и state machine;
- базовая moderation/validation;
- генерация текста гадания;
- TTS;
- сборка ответа с артефактами;
- симулятор механики;
- хранение истории шагов;
- минимальная админка или операторский CLI;
- тестовый контур распознавания карт.

### Можно отложить

- платежи;
- многоязычность;
- красивую публичную веб-панель;
- сложные пользовательские профили;
- продвинутую аналитику;
- масштабирование на много роботов;
- идеально автоматический recovery всех железных ошибок.

## Самый быстрый план проверки гипотезы

### Этап 0. Software-only mock

Цель: доказать, что пользовательский опыт и orchestration живут без механики.

Делаем:

- Telegram-бот;
- очередь;
- генерацию текста;
- TTS;
- подстановку карт из моков;
- сборку итогового ответа в Telegram.

Критерий успеха:

Пользователь получает правдоподобный полный результат без участия физической установки.

### Этап 1. Симулятор ритуала

Цель: проверить полный пайплайн длительной заявки.

Делаем:

- симулятор действий механики;
- временные задержки шагов;
- лог шагов;
- предзаписанное или программно собираемое видео.

Критерий успеха:

Видно, что весь flow переживает долгие операции и корректно отдает артефакты.

### Этап 2. Камера и media pipeline

Цель: проверить, как реально собирается видеоответ.

Делаем:

- подключение камеры;
- запись видео;
- наложение TTS;
- сборку финального ролика;
- контроль длительности и синхронизации.

Критерий успеха:

На выходе стабильно получается ролик приемлемого качества.

### Этап 3. Распознавание карт

Цель: понять, нужно ли нам сложное CV или достаточно контролируемой съемки.

Делаем:

- небольшой датасет собственных фото;
- классификацию карт;
- оценку точности;
- тесты на освещение, угол, тени и перекрытия.

Критерий успеха:

Распознавание работает достаточно стабильно в условиях конкретной установки.

### Этап 4. Интеграция механики

Цель: заменить симулятор на реальное устройство без перелома архитектуры.

Делаем:

- протокол команд к устройству;
- подтверждения выполнения;
- таймауты;
- обработку ошибок;
- ручной recovery.

Критерий успеха:

Один реальный цикл расклада проходит автоматически и предсказуемо.

### Этап 5. Закрытый alpha/beta

Цель: проверить устойчивость на реальных пользователях.

Делаем:

- лимитированный доступ;
- наблюдаемость;
- ручной контроль ошибок;
- сбор обратной связи.

Критерий успеха:

Понятно, какие ограничения нужно снять перед расширением.

## Основные риски

1. Распознавание карт может оказаться чувствительным к свету, бликам и положению карт.
2. Механика может ломаться чаще, чем приложение ожидает.
3. Синхронизация голоса, видео и физического ритуала может быть сложнее, чем кажется.
4. Prompt injection и нежелательные запросы потребуют не одной проверки, а нескольких слоев защиты.
5. Долгие задачи потребуют хорошей повторяемости и retry-логики.
6. Пользователю нужен прогнозируемый SLA: например, ответ за 1-3 минуты, а не "когда получится".
7. Для отладки почти наверняка понадобится режим ручного вмешательства.

## Вопросы, которые нужно закрыть до старта реализации

1. Что считается первой коммерчески полезной версией: текст + фото карт или именно полноценное видео с голосом?
2. Нужна ли живая запись камеры с реальным звуком, или для POC достаточно программно собрать итоговый ролик?
3. Сколько карт в первом раскладе и какой именно расклад берем за MVP?
4. Нужно ли распознавать карту камерой, если механика и так знает, какую карту выдала?
5. Где физически будет жить управляющая машина: рядом с роботом или в облаке?
6. Сколько одновременно может быть активных заявок на первом этапе?
7. Нужен ли оператор, который сможет вручную подтверждать сомнительные шаги?
8. Какие темы считаются запрещенными, кроме очевидных safety-ограничений?
9. Какой допустимый SLA для ответа пользователю?
10. Нужно ли хранить видео и фото долго, или это одноразовые артефакты?

## Мое текущее предложение

Не начинать с "настоящего робота".

Сначала собрать вертикальный software slice:

- Telegram -> queue -> validation -> LLM -> TTS -> simulated cards -> final response.

После этого добавлять по одному реальному риску:

1. media pipeline;
2. card recognition;
3. hardware adapter;
4. real device.

Такой порядок дает максимальную скорость проверки гипотезы и не блокирует нас механикой.
