# AGENTS.md

Руководство для LLM-агентов, которые работают в репозитории `tarobot`.

Цель файла:

- быстро вводить нового агента в контекст;
- не терять важные решения между чатами;
- сделать документацию главным местом хранения знаний, а не историю диалогов.

## Зачем этот репозиторий

`Tarobot` — сервис автоматизированного гадания на картах Таро.

В долгосрочной версии в системе будут:

- программная часть;
- механическая часть;
- камера и photo recognition;
- генерация текста гадания;
- TTS / голос;
- внешние интерфейсы, включая Telegram.

Сейчас проект на стадии `POC`.

`POC` здесь означает: быстро проверить, что ключевая идея вообще работает технически, не дожидаясь полной продуктовой упаковки.

## Что уже зафиксировано

На текущий момент приняты такие решения:

- Telegram не является текущим приоритетом и будет подключаться позже.
- Для MVP карты обязательны: нужно распознавание карт на фото.
- Фото является источником истины по раскладу.
- Механика может сообщать ожидаемое количество карт, но финальная проверка должна идти через фото.
- Для MVP лучше ложный отказ, чем ложное распознавание карты.
- Базовый расклад по умолчанию: 5 карт.
- Для тестов и отладки нужно поддержать расклад на 3 карты.
- Для раннего MVP достаточно артефактов `текст + аудио`.
- Видео пока не является обязательным условием POC.
- Ручной оператор в MVP не планируется.
- Архитектурная схема по размещению: `Raspberry Pi` рядом с механикой + облако или локальная машина для сервисной логики.
- Detection / count validation можно разрабатывать и проверять на обычных игральных картах.
- Финальное распознавание для Таро не должно опираться на угол карты и ранги/масти игральной колоды.
- Целевой путь для Таро: `whole-card recognition + upright/reversed + OCR fallback + layout interpreter`.

## Главные POC-gates

Сейчас проект должен в первую очередь пройти три проверки:

### `R1`

Система должна надежно находить все карты на фото и понимать, когда расклад невалиден.

### `R3`

Система должна корректно распознавать карту и уметь отказываться от результата при низкой уверенности.

### `R9`

Система должна выдавать аудио, которое звучит достаточно "живо" для демонстрации POC.

Пока эти три пункта не пройдены, все остальное считается вторым приоритетом.

## Ближайший рабочий порядок

Если нет новых указаний от пользователя, ориентируйтесь на такой порядок:

1. Фото-стенд, набор реальных фото, quality gate.
2. Detection / count validation по фото.
3. Recognition / orientation / confidence для Таро по всей карте, а не по corner-template схеме.
4. TTS-проверка на "живость".
5. Quality pass реальной LLM-генерации.
6. `Session` и `event log`.
7. Граница между основным сервисом и `hardware-agent`.
8. Реальный Raspberry Pi agent.
9. Telegram и остальная доставка.

## Что сейчас не приоритет

Пока не нужно оптимизировать в первую очередь:

- Telegram-интеграцию;
- полноценную очередь задач;
- финальный video pipeline;
- продвинутую админку;
- платежи;
- продуктовую обвязку.

## Правила для агентов

### 1. Не хранить ключевые решения только в чате

Если принято новое важное решение, обновите соответствующий файл в репозитории.

Примеры:

- изменились приоритеты или порядок работ — обновить [docs/iterations.md](/Users/avlsapozhkov/projects/tarobot/docs/iterations.md);
- появился новый важный риск — обновить [docs/risks.md](/Users/avlsapozhkov/projects/tarobot/docs/risks.md);
- изменились vision-допущения — обновить [docs/vision-mvp.md](/Users/avlsapozhkov/projects/tarobot/docs/vision-mvp.md);
- изменились проектные инварианты — обновить [README.md](/Users/avlsapozhkov/projects/tarobot/README.md) и при необходимости этот файл.

### 2. Фото важнее ожиданий механики

Если механика "думает", что разложила 5 карт, а фото показывает 4 или сомнительный overlap, доверяем фото.

### 3. Ложный отказ лучше ложного распознавания

Нельзя отправлять пользователю уверенный ответ по неверно распознанным картам.

### 4. Debug artifacts обязательны

Для vision-related работы сохраняйте как можно больше полезных артефактов:

- исходное фото;
- фото с overlay;
- crops карт;
- confidence;
- reason codes;
- manifest / session logs.

Без них невозможно нормально разбирать сбои между чатами и сессиями.

### 5. Документация и issues — часть результата

Если агент сделал работу, но не обновил документацию и не зафиксировал новый контекст, работа считается частично незавершенной.

### 6. Пишите по-русски

Для этого репозитория проектные документы, issue texts и основная коммуникация внутри репозитория должны быть на русском, если пользователь не попросил иное.

## Базовые команды

Локальный прогон:

```bash
python3 main.py "Что меня ждет?" --cards 5
```

С воспроизводимым сидом:

```bash
python3 main.py "Стоит ли делать следующий шаг?" --cards 3 --seed 42
```

Без аудио:

```bash
python3 main.py "Какой сейчас лучший следующий шаг?" --cards 3 --silent-tts
```

Тесты:

```bash
python3 -m unittest discover -s tests -v
```
